# Coverage-Guided Fuzzing

> Automatically generate adversarial inputs that maximize code coverage, finding crashes and memory-safety bugs that static analysis and manual testing routinely miss.

## The Problem

Code review and SAST catch many classes of vulnerability, but they reason about code structure, not runtime behavior. Bugs that only manifest under specific input combinations (malformed lengths, boundary values, unexpected encoding sequences, deeply nested structures) survive review and ship.

Manual testing explores too small a slice of the input space to be reliable. A test suite covering 80% of lines can still miss the exact byte sequence that triggers a heap overflow.

Real-world consequences:

- **Heartbleed (CVE-2014-0160)**: a bounds-check omission in OpenSSL's TLS heartbeat handler that exposed up to 64 KB of server memory per request. Present for two years before discovery.
- **ImageMagick "ImageTragick" (CVE-2016-3714)**: unsafe handling of specially crafted image files allowed remote code execution. Fuzz testing of the image parser would have been a natural fit.
- **LibPNG, libjpeg, expat**: all have shipped CVEs that fuzz testing caught in later audits.

If the code parses input (files, network data, environment variables, user-supplied strings), it has a surface area that manual testing cannot adequately cover.

## The Solution

### Overview

Coverage-guided fuzzing instruments the target binary so that each test run reports which edges of the control-flow graph it exercised. The fuzzer keeps inputs that reach new edges and discards those that do not. Over time, the corpus evolves toward unexplored corners of the code. When a run causes a crash, hang, or sanitizer violation, the input is saved as a finding.

The loop is: mutate → execute → measure coverage → keep if new → repeat.

This is more effective than random fuzzing because the feedback signal steers mutation energy toward code that has not been tested, rather than hammering already-covered paths.

### Core Concepts

| Concept | What it means |
|---|---|
| Fuzz harness | A thin wrapper that reads fuzzer-generated bytes and feeds them to the function under test |
| Seed corpus | A set of valid, representative inputs the fuzzer starts mutating from |
| Sanitizers | Compiler flags that instrument the binary to detect memory errors that would not crash normally |
| Coverage feedback | Edge-transition counts reported back to the fuzzer after each run |
| Crash reproduction | Re-running the exact input that caused a crash to confirm and minimize it |

### Implementation

#### Step 1: Identify the target

Fuzzing delivers the most value on code that:

- parses external input (files, network packets, API payloads, environment variables)
- handles variable-length data or complex encoding
- contains manual memory management (C, C++)
- is a critical dependency with a wide deployment footprint

Start with the entry points closest to untrusted data.

#### Step 2: Write a fuzz harness

A harness is a minimal program that accepts a byte buffer from the fuzzer and passes it to the target function. Keep the harness thin (harness logic is not tested by the fuzzer).

```c
// fuzz/fuzz_parse.c
#include <stdint.h>
#include <stddef.h>
#include "parser.h"

int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
    // Wrap in a guard to avoid OOM from intentionally huge inputs
    if (size > 65536) return 0;

    parse_input(data, size);
    return 0;
}
```

The `LLVMFuzzerTestOneInput` signature is understood by both AFL++ (via its libFuzzer compatibility mode) and libFuzzer, making the harness portable.

For targets that expect a file path rather than a byte buffer:

```c
int main(int argc, char **argv) {
    if (argc < 2) return 1;
    FILE *f = fopen(argv[1], "rb");
    if (!f) return 1;
    // read and process...
    fclose(f);
    return 0;
}
```

AFL++ passes the generated file path via `@@` in the command line.

#### Step 3: Compile with instrumentation and sanitizers

```bash
# AFL++ with AddressSanitizer and UndefinedBehaviorSanitizer
AFL_USE_ASAN=1 AFL_USE_UBSAN=1 afl-clang-fast \
  -o fuzz/fuzz_parse \
  fuzz/fuzz_parse.c src/parser.c \
  -I include/

# libFuzzer equivalent
clang -fsanitize=fuzzer,address,undefined \
  -o fuzz/fuzz_parse \
  fuzz/fuzz_parse.c src/parser.c \
  -I include/
```

Always enable at least **AddressSanitizer (ASan)**. Without it, many heap overflows and use-after-free bugs produce silent memory corruption rather than a crash, and the fuzzer will never see them as findings.

#### Step 4: Prepare a seed corpus

The seed corpus is the starting point for mutation. Seeds should be:

- **Valid**: the target should process them without immediately returning an error, so the fuzzer explores the interesting parsing paths
- **Diverse**: cover different code paths (different message types, different file variants, edge-case sizes)
- **Small**: smaller inputs mutate more efficiently

```bash
mkdir -p corpus/
cp test/fixtures/*.png corpus/        # real test files work well as seeds
echo '{"key":"value"}' > corpus/seed_json.txt
```

Minimize a large corpus before a long run:

```bash
afl-cmin -i raw_corpus/ -o corpus/ -- ./fuzz/fuzz_parse @@
```

#### Step 5: Run the fuzzer

```bash
mkdir -p findings/

# -t: per-run timeout in ms; -m: memory limit in MB
AFL_NO_UI=1 afl-fuzz \
  -i corpus/ \
  -o findings/ \
  -t 5000 \
  -m 512 \
  -- ./fuzz/fuzz_parse @@
```

For CI environments, run with a time limit:

```bash
timeout 3600 afl-fuzz -i corpus/ -o findings/ -- ./fuzz/fuzz_parse @@ || true
```

Watch progress:

```bash
afl-whatsup findings/   # summary across parallel instances
```

#### Step 6: Triage findings

```bash
# List crashes
ls findings/crashes/

# Reproduce a crash
./fuzz/fuzz_parse findings/crashes/id:000000,sig:06,...

# Minimize to smallest reproducer
afl-tmin -i findings/crashes/id:000000 -o minimal_crash -- ./fuzz/fuzz_parse @@

# Run under a debugger
gdb --args ./fuzz/fuzz_parse minimal_crash
```

Every crash should be triaged to determine:

1. Is it a true positive or a harness bug?
2. What is the root cause (overflow, NULL deref, use-after-free)?
3. Is it exploitable, or just a crasher?

Document findings with the minimal reproducer, sanitizer output, and a severity assessment.

## Code Examples

### Bad Practice (Vulnerable)

```c
// parser.c — no length validation before copy
void parse_header(const uint8_t *data, size_t size) {
    char name[64];
    // Copies until null byte — no bounds check against 'size' or array length
    strcpy(name, (const char *)data);
    process_name(name);
}
```

**Why this is problematic:**

- `strcpy` copies until a null byte regardless of destination capacity
- An input with no null byte in the first 64 bytes overflows `name` into adjacent stack memory
- This is exploitable as a stack buffer overflow (CWE-121)
- A fuzzer with ASan enabled will find this within seconds of starting

### Good Practice (Secure)

```c
// parser.c — bounded copy with explicit null termination
void parse_header(const uint8_t *data, size_t size) {
    char name[64];
    size_t copy_len = size < sizeof(name) - 1 ? size : sizeof(name) - 1;
    memcpy(name, data, copy_len);
    name[copy_len] = '\0';
    process_name(name);
}
```

**Fuzz harness that would catch the original bug:**

```c
int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
    if (size == 0) return 0;
    parse_header(data, size);
    return 0;
}
```

When compiled with `AFL_USE_ASAN=1`, the fuzzer will generate an input without a null byte in the first 64 bytes, trigger the overflow, and save the crash with a full ASan stack trace pointing directly to the `strcpy` call.

## Benefits

- **Finds bugs that static analysis misses**: runtime behavior, input-dependent paths, and memory-safety violations are invisible to SAST
- **Scales with compute**: more CPU time and more parallel instances directly translate into broader coverage
- **Produces concrete reproducers**: every finding comes with a minimal input that reliably triggers the bug, making triage faster than a static alert
- **Catches regressions**: a growing corpus retested on each CI run detects when a refactor reintroduces a previously fixed bug

## Common Pitfalls

1. **No sanitizers**: the fuzzer only reports crashes and hangs by default. Without ASan, most memory-safety bugs produce silent corruption and are never surfaced. Always build with at least `AFL_USE_ASAN=1`.

2. **Trivial or empty seed corpus**: starting with an empty corpus forces the fuzzer to discover valid structure from scratch. Provide a handful of real, valid inputs to seed meaningful exploration.

3. **Corpus not persisted between runs**: running the fuzzer from scratch each time wastes all the coverage progress from prior runs. Cache the `queue/` directory between CI runs.

4. **Running fuzzing on every commit**: fuzzing needs minutes to hours to be meaningful. Put it on a schedule (nightly or weekly), not in the pull request gate.

5. **Ignoring timeouts and OOM findings**: a hang can indicate a denial-of-service vulnerability. Do not discard `findings/hangs/` without investigation.

6. **Harness that crashes on valid input**: if the harness itself rejects too many inputs early, the fuzzer cannot explore interesting paths. Validate that seeds process cleanly through the harness before starting a long run.

7. **Never updating the corpus**: as the target evolves, old seeds may no longer reach new code. Periodically review and refresh the corpus when major features change the parsing surface.

## When to Apply

- **Always**: any C or C++ code that parses untrusted input and ships as a product or library
- **Recommended**: input parsers in other languages (Python, Rust, Go) using language-specific fuzzers; any code handling file formats, network protocols, or serialization
- **Consider**: internal services and APIs where the attack surface is smaller but a single bypass could have high impact

## Framework/Language-Specific Guidance

### C / C++ with AFL++

```bash
AFL_USE_ASAN=1 afl-clang-fast -o target fuzz_harness.c target_lib.c
afl-fuzz -i corpus/ -o findings/ -- ./target @@
```

### Rust with cargo-fuzz

Rust's ownership model prevents many memory bugs, but logic errors and panics are still meaningful findings:

```bash
cargo install cargo-fuzz
cargo fuzz add fuzz_target_name
cargo fuzz run fuzz_target_name
```

Cargo-fuzz uses libFuzzer under the hood and integrates cleanly with `cargo test`.

### Python with Atheris

```python
import atheris
import sys
import io

def TestOneInput(data):
    fdp = atheris.FuzzedDataProvider(data)
    s = fdp.ConsumeUnicodeNoSurrogates(128)
    target_function(s)

atheris.Setup(sys.argv, TestOneInput)
atheris.Fuzz()
```

```bash
pip install atheris
python fuzz_target.py
```

### Go with go-fuzz / native fuzzing

Go 1.18+ has native fuzz support:

```go
func FuzzTargetFunction(f *testing.F) {
    f.Add([]byte("seed"))
    f.Fuzz(func(t *testing.T, data []byte) {
        TargetFunction(data)
    })
}
```

```bash
go test -fuzz=FuzzTargetFunction ./...
```

## Verification & Testing

### Manual Checks

- Confirm that a seed corpus exists and seeds process without errors through the harness
- Confirm that the instrumented binary was built with at least ASan enabled (`ASAN_OPTIONS=detect_leaks=0 ./target seed_file` should not immediately crash)
- Confirm that `findings/crashes/` is monitored and triaged after each fuzzing run
- Confirm the corpus is cached and grows between CI runs rather than restarting each time

### Automated Testing

```yaml
- name: Verify no crashes in fuzz findings
  run: |
    crash_count=$(ls findings/crashes/ 2>/dev/null | grep -vc README || echo 0)
    echo "Crashes found: $crash_count"
    test "$crash_count" -eq 0
```

### Security Scanning

- **AFL++**: primary fuzzer for C/C++ targets
- **cargo-fuzz**: Rust targets
- **Atheris**: Python targets
- **OSS-Fuzz**: free continuous fuzzing infrastructure for open-source projects; integrates AFL++ and libFuzzer
- **ClusterFuzz**: Google's distributed fuzzing platform, open-source and self-hostable

## Standards & Compliance

- **CWE-119** (Improper Restriction of Operations within Buffer Bounds) — coverage-guided fuzzing is among the most effective methods for finding buffer handling bugs in this family
- **CWE-20** (Improper Input Validation) — fuzzing systematically probes input validation logic across the full mutation space
- **OWASP Top 10 A03:2021 – Injection**: injection bugs often manifest as crashes or sanitizer violations under fuzzing
- **NIST SSDF (SP 800-218) PW.8**: "Test for Security of the Application Throughout the SDLC" — continuous fuzzing directly satisfies this practice
- **OWASP ASVS V5 (Validation, Sanitization and Encoding)**: fuzzing provides runtime evidence that validation code holds under adversarial input

## Case Studies

### Incident Example

The **Heartbleed bug (CVE-2014-0160)** in OpenSSL allowed an attacker to request up to 64 KB of server memory by sending a crafted TLS heartbeat message with a mismatched length field. A fuzz harness targeting the heartbeat handler with a mutating length field and ASan enabled would have triggered an out-of-bounds read and surfaced the bug before it shipped. OSS-Fuzz, which began fuzzing OpenSSL in 2017, has since found dozens of additional bugs in the library.

### Success Story

Google's **OSS-Fuzz** program has run continuous coverage-guided fuzzing against thousands of open-source projects since 2016. As of 2024 it has found over 10,000 bugs across projects including OpenSSL, FFmpeg, curl, SQLite, and the Linux kernel, the majority of which were memory-safety bugs that had survived code review and unit testing. Projects integrating OSS-Fuzz typically report that it finds bugs within days of a code change that SAST missed entirely.
